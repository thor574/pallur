# v0.1 - Ollama og Continue

Dette er dokumentasjon av den første lokale løsningen. Innholdet bygger på eksisterende
kiitos-ruff-notater i `kiitos.ruff.th/oppgaver/RUFF-KI-LOKAL-01-*`.

## Formål

Målet var en lukket og privat utviklerflate i VS Code:

```text
Visual Studio Code
`-- Continue
    `-- Ollama på 127.0.0.1:11434
        |-- qwen3.6
        |-- north-mini-code-1.0
        |-- qwen2.5-coder:1.5b
        `-- bge-m3
```

Denne løsningen var laget for daglig interaktiv hjelp i editoren. Kiitos-orkesteret skulle
fortsatt brukes til mer formelle og sporbare agentløp.

## Roller

| Rolle | Modell | Bruk |
|---|---|---|
| Samtale, analyse og plan | `qwen3.6` | Kodeforståelse, konsekvensanalyse, designvalg |
| Implementasjon | `north-mini-code-1.0` | Konkrete kodeendringer og testskriving |
| Automatisk fullføring | `qwen2.5-coder:1.5b` | Små kodeforslag mens man skriver |
| Lokal semantisk indeks | `bge-m3` | Embeddings og lokal kodebaseinnhenting |

## Lokal låsing

Ruff-notatet anbefalte at Ollama bindes til loopback og at skyfunksjoner ikke brukes:

```text
OLLAMA_NO_CLOUD=1
OLLAMA_HOST=127.0.0.1:11434
OLLAMA_CONTEXT_LENGTH=32768
OLLAMA_NUM_PARALLEL=1
OLLAMA_MAX_LOADED_MODELS=2
OLLAMA_KEEP_ALIVE=30m
```

Kontroller etter oppstart:

```powershell
Invoke-RestMethod http://127.0.0.1:11434/api/tags

Get-NetTCPConnection -LocalPort 11434 -State Listen |
    Select-Object LocalAddress, LocalPort, OwningProcess
```

For lukket lokal drift skal lytteadresse være `127.0.0.1` eller tilsvarende lokal loopback.

## Continue-oppsett

Continue-konfigurasjonen fra ruffen bruker:

- `apiBase: http://127.0.0.1:11434`
- store modeller med rollene `chat`, `edit` og `apply`
- liten Qwen Coder-modell med rollen `autocomplete`
- `bge-m3` med rollen `embed`
- ingen eksterne modelltilbydere, API-nøkler, eksterne MCP-tjenere eller eksterne datakilder

Grunnfunksjoner som skulle prøves først:

1. automatisk fullføring
2. samtale om valgt kode
3. redigering med synlig diff

Fri agentmodus skulle vente til grunnoppsettet var stabilt.

## Arbeidsmoduser

Ruffen har et PowerShell-skript for kontrollert modellvarming:

```powershell
.\RUFF-KI-LOKAL-01-ollama-arbeidsmodus.ps1 -Modus interaktiv
.\RUFF-KI-LOKAL-01-ollama-arbeidsmodus.ps1 -Modus implementasjon
.\RUFF-KI-LOKAL-01-ollama-arbeidsmodus.ps1 -Modus orkester
.\RUFF-KI-LOKAL-01-ollama-arbeidsmodus.ps1 -Modus stopp
.\RUFF-KI-LOKAL-01-ollama-arbeidsmodus.ps1 -Modus status
```

Normalregel:

```text
Interaktiv utvikling:
qwen3.6 + qwen2.5-coder:1.5b

Konkret implementasjonsøkt:
north-mini-code-1.0 + qwen2.5-coder:1.5b

Tung orkesterkjøring:
Continue-fullføring av, orkesterets modellsekvens aktiv
```

## Erfaring

Brukeropplevelsen var at løsningen virket, men var treg. Sannsynlig forklaring er at store
modeller og kontekst hovedsakelig konkurrerte om systemminne, CPU og sideveksling, med
begrenset VRAM-rom på maskinen.

Dette er ikke en feil i arkitekturen, men viser at `v0.1` manglet tydelig måling og styring
av modellstørrelse, kontekst, varm modell, GPU-avlasting og samtidighet.

## Når dette sporet fortsatt er nyttig

Ollama og Continue er fortsatt et mulig spor når målet er:

- VS Code-integrasjon
- enkel lokal modellserver
- lokal autocomplete
- lokal embeddingmodell
- lite behov for separat terminalagent

Det bør ikke brukes som endelig sammenligningsgrunnlag før kaldstart, varmstart, tokenhastighet,
RAM, VRAM og sideveksling er målt på samme oppgave som `v1`.
